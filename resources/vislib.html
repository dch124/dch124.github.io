VISLIB - Vision Group Resource Library

This page describes the software resources that are contained in the VISLIB user space. Other school resources are available in xtools. PS. I'm still working on this...

Introduction

VISLIB is a suite of resources for use by the Leeds University Vision Group members. It lives in the user space ~vislib on both the DCRE and SGI networks. It contains applications, tools, libraries, source code, information and data which is (or may one day be, or may once have been) of use to Vision Group members. VISLIB is maintained collectively by group members.

Organization

VISLIB has user space on both DCRE and the SGI network, both as ~vislib. The DCRE account currently has 130Mb and the SGI account has 40Mb. The structure of both accounts is very similar (and in many places identical - by use of symbolic links); however, the DCRE account is geared towards IRIX 5.3/IRIX 6 and the SGI account is geared towards IRIX 4.

Both accounts has the following structure:

~vislib ----- src
          |-- bin
          |-- lib
          |-- include
          |-- tmp
          |-- demos
          |-- data
          |-- docs
          |-- man
          --- WWW

The directories shown above have the following uses:

src is for source code (and associated materials) of applications, tools and libraries. There is a separate subdirectory for each app/tool/lib, and the general idea is that each app/tool/lib should `live' here, and be compiled here, and the bin, lib and include directories should just contain symbolic links to the relevant executables/libraries/header files.

bin is for executables (IRIX 5/6 for DCRE, IRIX 4 for SGI). These either exist directly in bin or, more commonly, as symbolic links to files under src. bin is unstructured (no, or very few subdirectories) so that users can easily access executables simply by putting ~vislib/bin in their search path. This has the side effect of making the directory very crowded and unreadable. It should, however, be possible to work out what a particular executable is by looking as what its symbolic link is (using ls -l). For example if I notice that mr_slicer is a link to ../src/mrutils/process_utils/slicer, I know that it must be some kind of magnetic resonance image utility.

lib is for stand alone (.a) and shared (.so) libraries. Again these are mostly symbilic links to the physical files under the src directory. Libraries are also generally OS specific, so expect differences between the DCRE and SGI versions of this directory.

include is for header (.h) files. Mostly symbolic links again.

tmp is a temporary directory for newly-installed stuff. The idea is that if you want to download something and try it out you can put it in here. When you've evaluated whether it's any use or not you can either delete it or install it properly under src, document it (add it to the list on this page, and make README files), and tell everyone that it's available and how useful it is.

archive is for old stuff that I'm not sure what it's for or if anyone uses it.

demos contains demos of the work of Vision Group members. This is to allow people's work to be shown to visitors even if they're not there.

data is for vision-related data. Qualifying categories are images, volumetric data, video sequences, etc etc. Most data will be specific to one person and as such should be kept in their own user space. Stuff in here is either shared data or data that's too big for people's own user space.

docs is for useful documents or information. This is not for documentation of the apps/libs kept in VISLIB - this should be kept with the source code, as this is where users would naturally look for it. In fact, I don't think there will be much stuff in here at all, since most stuff will be in WWW.

man is for man pages. Again, not many of these.

WWW is for any web stuff (including this page, and the Vision Group home page).

Every directory should have a README or CONTENTS file included which gives a short description of what the directory contains, and more importantly, who owns or maintains it. This is essential in order to keep VISLIB in some kind of order, as the biggest problem with tidying it up has definitely been finding out who uses what.

Using VISLIB Utilities

To give yourself easier access to the utils in VISLIB, you can set up your environment in certain ways.

  • Set your PATH variable to include ~vislib/bin. Do this using setenv PATH ~vislib/bin:$PATH Add the above line to your .cshrc file for a more permanent change.
  • Include ~vislib/man in your search for man pages. Don't know how to do this - if anyone else does, please fill in here.

Contents

There now follows a list of what's currently installed/available in VISLIB, along with a few brief descriptions here and there.

Info/Docs

Rosenfeld's Computer Vision Bibliographies

docs/Rosenfeld/*

These are comprehensive bibliographies of computer vision papers published year by year. They're also available on the Web.

Applications

Image/Movie Format Conversion Utilities

bin/{*2*, *to*}

Convert various image and movie formats. There are loads more of these in ~xtools/pbmplus.

MRI Viewer and Utilities

bin/mr_*

Used to view (mr_viewer) and manipulate (the others) MR volume images held in data/mrimages. The source for the viewer is closely guarded by Andy Bulpitt. The source for the utils is in src/mrutils/view_utils.

Libraries

Lapack and Lapack++ matrix/linear algebra libraries

lib/{liblapack.a, liblapack++.a, libblas++.a}, include/lapack++.h

Very comprehensive matrix classes, using the BLAS fast algebra routines. If you read the user manual, it sounds really impressive. If you want to use it, ask Tony Heap for help - he has a little experience in it. Try reading the user manual.

There is a simple test program in ~vislib/src/lapack++/vislib_tests which shows you how to use lapack++. The associated makefile shows how to compile.

Octave

lib/{liboctave.a, libcruft.a}, include/octave/*.h

Some maths package. No idea what it does. If you know, please fill it in here and say whether it's any good.

AMB's home-grown matrix class library

lib/libmatrix.a, include/{NagMatrix.h, NagVector.h}

Home-grown C++ matrix and vector classes, written by Adam Baumberg. Used by his image classes, but also useable in their own right.

AJH's home-grown matrix class library

lib/libajhmatrix.a, include/{ajhmatrix.h}

Tony Heap's matrix class. For example code see /src/ajhmatrix.

AMB's home-grown image class library

lib/libis.a, include/image_class/*.h

Home-grown image classes, written by Adam Baumberg. Handle all aspects of image manipulation, including capture from Indycam/external camera/MPEG stream, processing (thresholding, filters, differencing etc) and output to screen. Copes with full colour, black and white, YUV - you name it, it does it. All that's needed is some documentation and a few sample programs that new users could take and easily hack about with.

Report Writing Utilities

Well, there aren't any yet. We wondered whether letter/report templates would be useful. What we do have is a nice gzipped eps of the Vision Group Logo.

Demos

Handtracker Demo

Tony Heap's hand tracker software for tracking hands in 2D and 3D. Look at the README files for more info, or look at the Hand Tracker Web site.

Pedestrian Behaviour Demos

Neil Johnson's demos of pedestrian typicality assessment and pedestrian trajectory prediction. See the README file for more info.

Data

Have a look and find out for yourself! At the moment there are a few old movies of cars, and some MR images of hands.

Maintenance

VISLIB can and should be maintained by all members of the Vision Group. If things are out of date, feel free to update them. If you're going to delete anything, mail the group first to check no-one uses it. If you move things around or update them or fix them, mail the group afterwards to let them know you've done it.

Adding new stuff

VISLIB is supposed to provide resources which could be of general use to the group, so if you've used something which you think other people would benefit from, such as an interface builder or some nice C++ class for handling 3D images or whatever, put it in here. Also, if you have any good demos of your work they can go in the demos directory, or you could alternatvely put them on your homepage.

In order to keep some form of order, try to observe the following guidelines:

  • If you're downloading something to try it out, you can put it initially in the tmp directory. If it turns out to be naff, delete it.
  • When you install it proper, refer to the use of directories described above to make sure you put it in the right place.
  • gzip large data files, and strip and gzexe large executables to save space.
  • Make sure any required symbolic links from the bin, lib and include directories are set up. If executables have particularly common names it might be an idea to give them a prefix to put them in context, eg. all the MRI utils in bin are prefixed with mr_, so flip has become mr_flip, which is more helpful.
  • Put a README or CONTENTS file in every directory you create, saying what it it and who installed it (ie. you).
  • Put a short description of it in the Contents section above. If you downloaded it, include any links to wherever you got it from. If it's homegrown, and you're feeling energetic, write a Web page describing it.
  • Mail the group members to tell everyone about the great new thing you've installed!

If disk space is running short you can apply to get some more. This is best done through official channels (go through David or Roger).

Any major changes to the structure of the user space should probably be discussed with everyone else first.
